# Proxy And Adapters

This file covers the local cache-registry, the tool adapters, and the serve/runtime internals behind them.

## Core shape

- `cache-registry` is the proxy.
- `cache-registry` is the standalone operator-facing proxy command.
- Adapter commands either start that proxy in the background or hand off to a small helper.
- `src/serve/**` is the real proxy runtime.
- `src/serve/engines/**` is the incremental engine-boundary home for protocol invariants that should not live in route handlers.
- `src/proxy/**` is the CLI-side wrapper around tag derivation, child command launch, and proxy startup.

Dual-proxy contention E2E treats the shutdown active-upload loser as an accepted outcome, not as a timeout or permanent drop. The harness reports accepted contention separately, fails on forced proxy stops, records remote-tag `pending` distinctly from hits and misses, and verifies final visibility without pinning the tag to a specific winning proxy publish id.

Telemetry direction note: web ADR 0008 makes `cache_session_summary.v2` plus `cache_rollups` the shared product telemetry path for archive, proxy, adapters, benchmarks, and first-class MCP readers, while raw run JSONL should persist as compressed object-storage event files for replay/debug. CLI producers should converge on bounded backend API operation/status/retry counts, provider-neutral storage fields, optional runner-headroom snapshots, bounded miss/degradation reason counts, negative-cache evidence, and explicit product-behavior classification; `/metrics` is legacy archive compatibility only and should not receive new bottleneck fields. The customer-facing diagnostics direction is CLI/TUI/Actions-first, with the TUI showing customer-owned general-purpose insights such as setup, tag/naming, bad cache state, customer-controlled misses, and fixable runner pressure. BoringCache-owned backend API, storage, lifecycle, proxy, release, and benchmark regression evidence stays behind JSON/debug/admin/internal MCP surfaces, with customer output limited to a clear service-side diagnosis and run id. MCP should consume the same summaries, ownership tags, evidence references, and bounded raw-event slices instead of asking an LLM to mine logs directly.

Benchmark alignment note: benchmark artifacts should emit canonical `run_uid`, product refs, lane/scenario, sample validity, reporting mode, and normalized session-summary fields. `benchmarks/scripts/launch-proof.rb` is the release-path gate for product refs, cache mode/lane, sample classification, Docker OCI counters, attached `cache_session_summary` evidence, and the launch matrix of tool/surface/scenario paths before benchmark output can support launch claims. The matrix distinguishes fresh cold runs, same-runner reruns, fresh-runner reruns, fresh-after-purge reruns, and Docker same-ref/same-alias proof so one warm sample cannot stand in for coverage. Rails should treat benchmark comparison code as an artifact adapter and delegate diagnosis/severity to the shared cache-health review output. CLI/action producers should avoid bespoke benchmark-only field names where a `cache_session_summary.v2` section already exists.

Telemetry vocabulary rule: any CLI field named `tool` or `adapter` that goes into cache rollups, missed-key records, session rows, metadata hints, status snapshots, `cache_session_summary`, JSONL observability, benchmark artifacts, or MCP-facing summaries must use the Rails canonical rollup vocabulary: `runtime`, `turborepo`, `nx`, `bazel`, `gradle`, `maven`, `sccache`, `gocache`, `oci`, or `archive`. User-facing adapter commands may remain `turbo`, `go`, and `docker`, but telemetry must canonicalize those as `turborepo`, `gocache`, and `oci`; only a separate field such as `adapter_command` should carry the command spelling if that distinction matters. Proxy metadata parsing treats `tool` and `adapter` as reserved keys: known aliases such as `turbo`, `go-cacheprog`, `docker-registry`, and `rust-sccache` are accepted but normalized before session/rollup reporting.

## Current active internals

These are internal, but they are clearly active, not dead:

- `src/serve/runtime/**`
- `src/serve/engines/**`
- `src/serve/http/**`
- `src/serve/cache_registry/**`
- `src/serve/state/**`
- `src/proxy/command.rs`
- `src/proxy/tags.rs`

OCI manifest reads must prove the referenced descriptor set before serving the manifest to clients like BuildKit. If the verified blob download-url response reports any missing descriptors, the proxy treats the manifest ref as a miss and does not populate the manifest cache or blob locator.

## Thin or weakly connected surfaces in this family

- `go-cacheprog` is supported, but only as an advanced helper behind `go` or manual `GOCACHEPROG` setups
- `bazel`, `gradle`, and `maven` are supported, but thinner wrappers than `docker`, `turbo`, `nx`, and `sccache`
