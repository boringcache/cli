# Turbo / Nx Remote Cache Mistake Ledger

This ledger captures source-backed guardrails for the Turbo and Nx adapter pass. The source column is the authority; current BoringCache behavior only shows where to preserve or improve compatibility.

| Failure class | False assumption | Source-backed invariant | Owning layer | Guardrail test | Residual risk |
| --- | --- | --- | --- | --- | --- |
| Nx upload header treated as incidental | The local proxy can accept any `PUT /v1/cache/{hash}` body as long as auth and path match | The Nx custom remote cache OpenAPI marks `Content-Length` as a required upload header and keeps the archive body opaque | `serve::engines::nx` artifact upload handling | Nx artifact round trip, duplicate upload conflict, and query seed uploads now include explicit upload `Content-Length`; local Rails-backed Nx E2E passed on 2026-04-22 | `HEAD`, terminal-output, and query routes remain compatibility additions outside the current Nx OpenAPI |
| Nx artifact overwrite accepted silently | A repeated `PUT` for the same task hash can replace the cached archive because generic KV accepts overwrites | The Nx custom remote cache OpenAPI defines `409` as the response when an upload cannot override an existing record | `serve::engines::nx` plus KV PUT options | `test_nx_artifact_put_returns_conflict_for_existing_record` proves duplicate artifact uploads return `409` and preserve the first payload | Cross-process races still depend on backend publish visibility; this slice protects the local proxy's pending/flushing/published state |
| Nx archive metadata collides with root output directories | Because the self-hosted server treats artifacts as opaque, every uploaded tar member is safe to replay exactly as sent | Nx 22.x stores task outputs in a gzip tar and appends root metadata files named `terminalOutput` and `code`; its reader branches on those root paths before checking entry type, so a root output directory named `code` can panic on warm restore | `serve::engines::nx` artifact upload compatibility shim | `nx_artifact_normalizer_strips_reserved_root_directories_only` proves the proxy drops only exact root directory entries named `code` or `terminalOutput` while preserving nested outputs and metadata files | This is a fail-open compatibility guard for the observed Nx 22.x archive reader; keep the public contract at the HTTP API boundary and remove/widen only with source-backed Nx changes |
| Nx terminal output misses untested | Terminal output objects are secondary and can inherit artifact behavior without explicit proof | Nx task replay depends on terminal output as part of cached task behavior; BoringCache's supported route must preserve miss status and empty `HEAD` bodies | `serve::engines::nx` terminal-output route over shared KV lookup | `test_nx_terminal_output_get_and_head_misses_return_not_found` proves `GET` and `HEAD` misses return `404`; local Rails-backed Nx E2E proved warm replay after `nx reset` on 2026-04-22 | Current Nx OpenAPI still only documents artifact upload/download; keep terminal-output, `HEAD`, and query routes tied to observed tool traffic before widening the official claim |
| Turborepo API shape hidden in route glue | Turbo status, artifact, query, and event behavior can remain incidental HTTP routing | The Turborepo Remote Cache API owns bearer auth, status response, artifact hash format, upload `Content-Length`, metadata header validation, batch query shape, and event payload validation | `serve::engines::turborepo` plus KV object substrate | Turbo status/auth, artifact hit/miss, invalid upload header, query metadata, and events tests prove the current supported API shape after the engine move | Team scoping stays request-transparent; add product modeling only with a source-backed multi-tenant design |

## Research Sources

- Nx self-hosted cache docs and Nx 22.6.1 `http_remote_cache.rs`, rechecked 2026-04-29: custom cache servers use `PUT` and `GET` under `/v1/cache/{hash}`; uploads require `Content-Length`, successful uploads are `200`, misses are `404`, auth failures are `401`/`403`, and duplicate records are `409`. The public docs keep the artifact body opaque; the 22.6.1 implementation currently gzip-tars task outputs and appends root `terminalOutput` and `code` metadata files.
- Turborepo Remote Cache API: artifact, status, query, and events routes remain the Turbo source of truth; artifact uploads require hex hashes, `Content-Length`, base64 tags when present, bounded client metadata headers, and JSON error bodies.
